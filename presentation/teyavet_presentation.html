What Runs Where — AWS Architecture

  • EKS 1.31 — managed node group (t3.medium, private subnets)
  • RDS MySQL 8 — private subnet only, no public access

Where Are Docker Images Built?
push to master
→
GitHub Actions runner
→
AWS ECR
  • 3 images: backend · frontend · db
  • Frontend: multi-stage (Node build → Nginx serve)
  • Backend: non-root appuser for least privilege
  • ECR lifecycle: keep last 10 tags, expire untagged after 1 day
Responsible Hosting — Security Controls
  • No static AWS keys — GitHub OIDC → STS short-lived tokens
  • RDS in private subnet — unreachable from internet
  • ECR image scanning on every push
  • Non-root containers — Flask runs as appuser
  • Secrets in GitHub Secrets — injected at runtime, never in code
  • Security groups: RDS only accepts MySQL from EKS nodes

Kubernetes — Cluster Internals
EKS Architecture — Pods, Networking & Permissions
Traffic Routing
  • ALB — internet-facing, HTTP :80
  • Path rules in teyavet-ingress:
/users /pets /appointments …
→ backend-service :5000
/ (catch-all)
→ frontend-service :80
ClusterIP services target-type: ip
AWS Load Balancer Controller
  • Runs in kube-system namespace
  • Installed via Helm on every deploy
  • Watches Ingress objects → provisions ALB
  • IRSA: own IAM role scoped to ALB/EC2 APIs only
backend — 2 replicas (Flask :5000)
  • Image from ECR — tag injected at deploy time
  • Security: runAsUser=1000, allowPrivilegeEscalation=false
  • Resources: 250m CPU / 256Mi → 500m / 512Mi
  • Liveness + readiness probes on /health
  • Rolling update: maxUnavailable=0 (zero-downtime)
frontend — 2 replicas (Nginx :80)
  • Multi-stage image: Node build → Nginx static serve
  • No secrets needed — pure static SPA
  • Resources: 100m / 64Mi → 200m / 128Mi
Config & Secrets
  • ConfigMap backend-config — DB_HOST, CORS_ORIGINS, FLASK_APP (non-sensitive)
  • Secret backend-secrets — DB_PASSWORD, JWT_SECRET_KEY
  • DB migration job runs schema.sql + seed_data.sql before pods start
IAM Roles & Permissions
  • EKS Node Role → AmazonEC2ContainerRegistryReadOnly — nodes pull ECR images (no imagePullSecrets needed)
  • LBC Role (IRSA) → create/manage ALBs, target groups — scoped to ELB + EC2 APIs
  • GitHub OIDC Role → AdministratorAccess — used by CI/CD to deploy, push to ECR, manage infra
  • EKS Cluster Role → AmazonEKSClusterPolicy
Security Summary
  • No imagePullSecrets — node IAM role handles ECR auth transparently
  • IRSA — LBC pod gets its own scoped role via service account annotation, not node role
  • K8s Secrets — sensitive values never in ConfigMap or image
  • Non-root pods — backend enforces runAsNonRoot
  • RDS unreachable — private subnet, SG allows only EKS node IPs
